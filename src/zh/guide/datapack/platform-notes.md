# 平台差异

Katton 面向 Fabric、NeoForge 和 Paper，但数据包开发者需要预期每个平台的能力上限不同。

## 快速对照

| 能力 | Fabric | NeoForge | Paper |
| --- | :---: | :---: | :---: |
| 服务端脚本 | 是 | 是 | 是 |
| 数据包调用 helper | 是 | 是 | 是 |
| 事件 | 是 | 是 | 是 |
| 数据包修改 | 是 | 是 | 是 |
| 自定义注册表 | 是 | 是 | 否 |
| 客户端脚本 | 是 | 是 | 否 |
| 客户端渲染 | 是 | 是 | 否 |
| 服务端到客户端脚本同步 | 是 | 是 | 否 |

## Fabric 与 NeoForge

如果你想使用完整 Katton 能力，选择 Fabric 或 NeoForge：注册表内容、客户端 HUD、渲染器、同步脚本包和模组级 API。

当你的数据包创意开始像一个真正的模组时，这两个平台更合适。

## Paper

如果你需要的是插件服上的服务端脚本，Paper 很合适。它适合命令风格行为、事件逻辑、数据包修改和 Bukkit 生态集成。

不要围绕自定义客户端内容设计 Paper 脚本。原版客户端无法从 Paper 服务端接收 Katton 定义的物品、方块、实体、模型或渲染器。

## 怎么选

如果迁移主要是替代命令逻辑，三个平台都可以。如果迁移会新增自定义内容或客户端视觉，目标应选 Fabric 或 NeoForge。
